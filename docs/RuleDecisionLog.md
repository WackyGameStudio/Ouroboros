## 3. 데이터 소유권

### 3.1 정적 설정과 런타임 복사

| 데이터 | 정적 원본 | 런타임 소유/실행 | 계약 |
| --- | --- | --- | --- |
| 플레이어 HP·이동·머리 공격·자석 | `OSPlayerBalanceData` | `OSSessionRuntimeState`, `OSPlayerHealth`, `OSPlayerController`, `OSHeadWeapon` | RuntimeState가 값을 소유하고 각 시스템은 해당 규칙만 적용 |
| 몸통·역할·절단·폭발 | `OSBodyBalanceData` | `OSBodyChain`, 역할 관리자 4종, `OSExplosionController` | 체인 순서·예약·소비의 단일 소유자는 OSBodyChain |
| 적·정예·보스·상한 | `OSEncounterBalanceData` | `OSEnemyController`, `OSEnemyRegistry`, `OSWaveDirector` | Registry는 활성 목록, Enemy는 개체 상태, Wave는 생성 결정 소유 |
| 0~10분 웨이브 | `OSWaveScheduleData` | `OSWaveDirector` | 세션 시간에 따른 스폰 티켓과 특별 등장 소유 |
| 업그레이드 15종 | `OSUpgradeCatalog` | `OSSessionRuntimeState`, 레벨업 후보 생성기 | 카탈로그는 불변, 선택 결과는 RuntimeState에만 적용 |
| 세션 상태·시간·FixedUpdate 순서 | 없음/세션 설정 | `OSGameSessionController` | 상태와 `Time.timeScale`의 유일한 소유자 |
| Body/LevelUp 요청 | 없음 | `OSSelectionQueue` | Body 우선 직렬화의 유일한 소유자 |
| 물리 틱 후보 | 없음 | `OSCombatEventBuffer` | 콜백에서는 후보만 수집하고 결과는 SessionController가 확정 |
| UI 표시 | 이벤트/RuntimeState 읽기 | HUD와 선택 패널 | UI는 규칙 상태를 직접 수정하지 않음 |

### 3.2 식별자 규칙

- 데이터 ID는 소문자 `snake_case`를 사용하고 빌드 내에서 고유해야 한다.
- 런타임 개체 ID와 공격 이벤트 ID는 세션 안에서 단조 증가하며 데이터 ID와 구분한다.
- ID 중복은 부트 검증 오류이며 자동 보정하지 않는다.

## 7. 0~10분 웨이브 계약

| 시간 | 신규 요소 | 필수 목적 |
| ---: | --- | --- |
| 0~1분 | `enemy_chaser` | 빈 공간에서 이동·자동 공격 학습 |
| 1~2분 | `enemy_charger` 소량 | 직선 예고 회피 학습 |
| 2~3분 | `enemy_shooter` 소량 | 우선 표적과 투사체 동선 판단 |
| 3분 | `enemy_elite_accelerator` 1회 | 오라 군집 분리 또는 첫 폭발 |
| 3~4분 | 추적+돌진+사격 | 첫 혼합 압박 |
| 4~5분 | `enemy_splitter` 추가 | 밀도와 광역 가치 상승 |
| 5~6분 | 일반 적 4종 | 빌드 분기 검증 |
| 6분 | `enemy_elite_accelerator` 1회 | 손실 후 재성장 검증 |
| 6~8분 | 고밀도 혼합 | 활성 적 150 전후 판독성 시험 |
| 8~9분 | 돌진·사격 비율 상승 | 단순 원형 회피 방지 |
| 9~10분 | 보스 예고 웨이브 | HP·몸통·폭발 보존 판단 |
| 10분 이후 | `boss_swarm_core` + 제한 소환 | 전체 빌드 검증 |

- 일반 적 HP 배율 초기값은 분당 `1.12^minute`, 생성 압력은 `1.15^minute`이다.
- 3분·6분 정예와 9분 예고는 각 1회만 발생한다.
- 선택 상태에서는 웨이브 시간이 정지하고 재시작 시 스케줄 상태를 초기화한다.

## 8. 범위 등급

| 등급 | 포함 | 제외/후순위 |
| --- | --- | --- |
| P0 | 이동, 몸통, 절단, 폭발, 역할, 성장, 적, 웨이브, 보스, 결과, 재시작 | 없으면 코어 게임 또는 빌드가 성립하지 않는 항목 |
| P1 | 게임패드, 세부 옵션, 추가 가독성, 고급 프로파일 도구 | P0 게이트 통과 후 품질·편의 강화 |
| P2 | 영구 해금, 추가 맵, 일일 도전, 멀티플레이 | MVP 이후 별도 기획 |
| Risk Track | 커스텀 머리 목업과 ComfyUI 실제 API | 실패해도 P0를 막지 않으며 기본 캐릭터/목업으로 폴백 |

## 9. 변경 절차

1. 변경 대상이 `[확정]`, `[가설]`, `[제외]` 중 무엇인지 먼저 판정한다.
2. `[가설]` 변경은 `BalanceHypotheses.md`의 관찰 결과와 변경 이유를 남긴다.
3. 규칙·ID·소유권 변경은 설계서와 영향받는 구현 Step 및 테스트를 같은 작업에서 갱신한다.
4. 최신 결정만 본문에 유지하고 과거 결정은 구현 규칙으로 남기지 않는다.
